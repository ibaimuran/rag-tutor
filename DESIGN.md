# RAG Tutor 设计文档

## 1. 项目概述

RAG Tutor 是一个基于 **Bloom 2-Sigma 掌握式学习理论**的智能一对一私教系统，结合 **DeepSeek 大语言模型**、**RAG（检索增强生成）** 和 **BKT（贝叶斯知识追踪）** 三大核心技术，为学生提供自适应、个性化的学习体验。

### 1.1 核心设计理念

- **掌握式学习**：不按固定时间表推进，而是基于学生对每个知识点的实际掌握程度动态调整学习路径
- **苏格拉底式教学**：AI 导师通过提问引导学生自主思考，而非直接灌输答案
- **数据驱动决策**：BKT 模型用概率论方法量化学生对每个知识点的掌握程度，区分"真知道"和"蒙对的"

## 3. BKT（贝叶斯知识追踪）模型

### 3.1 模型原理

BKT 将学生的知识状态建模为一个**隐马尔可夫模型 (HMM)**：

- **隐状态**：学生是否真正掌握了知识点 (Know / Not Know)
- **观测**：学生每次答题结果 (Correct / Incorrect)

### 3.2 四个核心参数

| 参数 | 含义 | 默认值 | 说明 |
|------|------|--------|------|
| P(L₀) | 初始掌握概率 | 0.50 | 学习前学生已掌握该知识点的概率 |
| P(T) | 学习转移概率 | 0.20 | 从未掌握到掌握的转移概率（学习效果） |
| P(G) | 猜测概率 | 0.15 | 未掌握但蒙对的概率 |
| P(S) | 失误概率 | 0.10 | 已掌握但答错的概率（粗心等） |

### 3.3 贝叶斯更新公式

每次观测后更新后验概率：

```
P(know | obs) = P(obs | know) * P(know) / P(obs)

其中：
- 答对时：P(obs|know) = 1-P(S), P(obs|¬know) = P(G)
- 答错时：P(obs|know) = P(S), P(obs|¬know) = 1-P(G)

更新后：P(know)new = P(know|obs) + (1-P(know|obs)) * P(T)
```

### 3.4 掌握度判定

| P(know) | 状态 | 系统行为 |
|---------|------|----------|
| >= 80% | mastered (已掌握) | 自动晋级下一知识点 |
| 60%-80% | learning (学习中) | 继续当前知识点的教学 |
| < 60% | needs_relearn (需重学) | 强制回到阅读模式重新学习 |

## 4. 教学状态机

### 4.1 状态转换

```
                    ┌──────────┐
                    │   IDLE   │ ← 初始状态 / 切换知识点
                    └────┬─────┘
                         │ 进入问答模式
                    ┌────▼─────┐
                    │DIAGNOSING│ ← 诊断阶段（1-2轮）
                    └────┬─────┘
                         │ 完成诊断
                    ┌────▼─────┐
                    │ TEACHING │ ← 教学阶段（3-4轮）
                    └────┬─────┘
                         │ 完成教学轮次
                    ┌────▼──────┐
                    │MASTERY    │ ← 掌握度检验
                    │_CHECK     │
                    └─┬──┬──┬──┘
                      │  │  │
              P>=80%  │  │  │  P<60%
              晋级    │  │  │  重学
              ┌───┐   │  │   ┌──────────┐
              │IDLE│←─┘  └──→│REMEDIATING│
              └───┘           └─────┬────┘
                                     │
                              P>=80% │  │ P<60%
                              晋级   │  │ 继续补救
                              ┌───┐  │
                              │IDLE│←─┘
                              └───┘
```

### 4.2 各阶段策略

| 阶段 | 策略 | 典型问题 |
|------|------|----------|
| DIAGNOSING | 探测先验知识，发现误解 | "你认为X是什么？请选出最符合的一项" |
| TEACHING | 苏格拉底式引导，从不直接给答案 | "如果改变这个条件，结果会怎样？" |
| MASTERY_CHECK | 综合性应用问题 | "请综合运用A和B的知识来解决..." |
| REMEDIATING | 聚焦薄弱点，换角度重新讲解 | "让我们从另一个角度来看这个问题..." |

## 5. RAG 管线（检索增强生成）

RAG 管线将 LLM 生成的知识点教材内容向量化存储，在教学对话中通过语义检索提供上下文，确保 AI 私教的回答基于课程内容。

### 5.1 数据流

```
LLM 生成的知识点教材内容
    │
    ▼
Embedder (BGE-large-zh-v1.5) ─── 将文本块转为768维向量
    │
    ▼
ChromaDB VectorStore ─── 持久化存储向量 + 元数据
    │
    ▼
Retriever ─── 根据知识点标题/描述进行语义检索（Top-K=5）
    │
    ▼
LLM Prompt ─── 检索结果作为上下文注入教学对话
```

### 5.2 嵌入模型

使用 `BAAI/bge-large-zh-v1.5`（中文优化，768 维），本地 CPU 推理。

## 6. 课程生成系统

### 6.1 主题驱动课程生成

1. 用户输入主题（如"初中化学"）
2. LLM 按照中国 K12 课程标准设计完整课程结构，必须覆盖该学科对应年级教材的全部章节
   - 例如"初中化学"：包含绪言 + 11 个单元约 160+ 个知识点
   - 章节标题使用教材标准名称，按教材顺序排列
3. 为每个知识点生成 500-800 字 MD 教材
4. 保存到文件系统 + 数据库 + ChromaDB
4. 建立向量索引

### 6.2 课程数据结构

```
Course (课程)
  ├── Chapter 1 (章节)
  │   ├── KnowledgePoint 1.1 (知识点)
  │   ├── KnowledgePoint 1.2
  │   └── ...
  ├── Chapter 2
  │   └── ...
  └── ...
```

知识点之间有前置依赖关系（prerequisites），确保学习路径有序。

## 7. 测验模块（Quiz）

### 7.1 设计思路

测验模块是对原问答模式的改进。原问答模式采用苏格拉底式对话，每次出一道题并要求学生解释答案，这种方式虽然深入但效率较低。测验模块改为：

- **批量出题**：每个知识点预生成 10 道选择题
- **逐题作答**：每次只显示一道题，答完自动跳到下一题
- **即时反馈**：每题作答后显示正误和正确答案
- **BKT 综合评估**：10 题全部完成后，BKT 模型综合所有答题记录评估掌握概率

### 7.2 测验流程

```
用户点击"问答"标签
    │
    ▼
系统问候："你好，我是你的知识点掌握程度检测助手"
    │
    ▼
显示第1题 → 用户点击选项 → 即时反馈 → 1.5s后自动跳转
    │
    ▼
显示第2题 → 用户点击选项 → 即时反馈 → 1.5s后自动跳转
    │
    ... (重复至第10题)
    │
    ▼
BKT 综合评估 → 展示掌握概率 + 掌握状态 + 答对统计
```

### 7.3 BKT 猜对检测

测验模式通过多题综合评估来区分"真知道"和"猜对"：

- **连续正确**：BKT P(know) 快速上升至 90%+
- **随机正确**：虽然答对部分题，但错误分布随机，BKT P(know) 收敛于 30-50%
- **全部错误**：P(know) 下降至 20% 以下，判定为未掌握

10 题的数据量足够 BKT 模型做出可靠的二分类判断。

## 11. 关键设计决策

### 10.1 为什么用 BKT 而不是简单的正确率？

简单的正确率（如 7/10 = 70%）无法区分以下情况：
- 学生 A：掌握了概念但粗心错了 3 题（真实掌握度 ~90%）
- 学生 B：没掌握但蒙对了 7 题（真实掌握度 ~30%）

BKT 通过 P(G) 和 P(S) 参数建模了猜测和失误，能从答题序列中推断真实掌握状态。

### 10.2 为什么每次只出一道题？

基于认知负荷理论，一次抛出多道题会增加学生的认知负担。逐题作答配合即时反馈，更符合掌握式学习的"小步快跑"原则。

### 10.3 为什么用 ChromaDB 而不是 FAISS？

ChromaDB 提供了开箱即用的持久化和 metadata 过滤，部署更简单。对于本项目的文档量级（几百到几千个 chunks），ChromaDB 的性能完全足够。
