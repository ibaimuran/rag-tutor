## 第4章 函数与模块

### 4.5 异常处理

#### 1. 核心概念

想象一下，你正在写一个程序，它需要用户输入一个数字。如果用户不小心输入了字母“abc”，程序会怎么样？它会直接崩溃，并显示一个红色的错误信息，然后停止运行。这就像一辆汽车在行驶中突然爆胎，直接抛锚在路中间，非常不友好。

**异常处理**，就是为你的程序准备一个“备用轮胎”和“安全气囊”。它允许你预测程序运行时可能出现的错误（比如用户输入错误、文件找不到、网络连接失败等），并提前写好“如果发生这种错误，就执行这段代码”的指令。这样，当错误发生时，程序不会直接崩溃，而是会优雅地处理这个错误，比如提示用户重新输入，或者记录下错误日志，然后继续运行下去。

简单来说，**异常处理**就是让程序在面对意外情况时，能够保持稳定和友好，而不是直接“死机”。

#### 2. 关键定义

-   **异常（Exception）**：程序在运行过程中发生的、会中断正常执行流程的意外事件。例如，除零操作（`ZeroDivisionError`）、访问不存在的列表索引（`IndexError`）、尝试打开一个不存在的文件（`FileNotFoundError`）等。
-   **抛出异常（Raise an Exception）**：当错误发生时，Python 解释器会创建一个异常对象，并“抛出”它，就像扔出一个信号弹，告诉程序：“出问题了！”
-   **捕获异常（Catch an Exception）**：程序使用 `try...except` 语句来“捕获”这个被抛出的异常信号，并执行预先准备好的处理代码。
-   **`try...except` 语句**：Python 中用于实现异常处理的核心语法结构。`try` 块中放置可能出错的代码，`except` 块中放置当特定异常发生时要执行的代码。
-   **`finally` 块**：一个可选的代码块，无论是否发生异常，其中的代码都会被执行。通常用于执行清理工作，比如关闭文件、释放网络连接等。

#### 3. 示例与应用

1.  **银行取款机**：你输入取款金额，如果金额大于你的余额，程序不会直接崩溃，而是会抛出一个“余额不足”的异常。ATM 机捕获这个异常后，会显示“余额不足，请重新输入”，而不是直接死机。

2.  **在线购物**：你提交订单后，程序需要连接支付网关。如果网络突然中断，程序会抛出一个“网络连接失败”的异常。购物网站捕获这个异常后，会提示你“支付失败，请稍后重试”，而不是直接显示一个空白页面。

3.  **文件读取**：你的程序需要读取一个配置文件。如果这个文件被用户误删了，程序会抛出一个 `FileNotFoundError` 异常。通过异常处理，程序可以捕获这个错误，并使用默认配置继续运行，而不是直接报错退出。

#### 4. 常见误区

-   **误区一：用异常处理代替正常的逻辑判断**
    -   **错误理解**：很多初学者会这样写代码：
        ```python
        try:
            number = int(input("请输入一个数字: "))
            print(10 / number)
        except:
            print("出错了")
        ```
        这虽然能工作，但非常不推荐。因为 `except` 会捕获所有类型的异常，包括你可能没预料到的错误（比如内存不足），导致你无法定位真正的问题。更好的做法是使用 `if` 语句进行前置判断：
        ```python
        number = input("请输入一个数字: ")
        if number.isdigit() and int(number) != 0:
            print(10 / int(number))
        else:
            print("输入无效，请输入一个非零数字")
        ```
    -   **正确理解**：异常处理应该用于处理那些**不可预测**或**难以通过前置判断完全避免**的错误（如网络问题、文件权限问题），而不是用于处理可以通过简单的 `if` 语句就能避免的逻辑错误。

-   **误区二：使用过于宽泛的 `except` 语句**
    -   **错误理解**：直接使用 `except:` 或 `except Exception:` 会捕获所有异常，导致你不知道具体发生了什么错误，难以调试。
    -   **正确理解**：应该尽量指定要捕获的**具体异常类型**，例如 `except ValueError:` 或 `except FileNotFoundError:`。这样，你就能针对不同的错误采取不同的处理方式，也让代码的意图更加清晰。
