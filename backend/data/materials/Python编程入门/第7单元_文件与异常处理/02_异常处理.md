# 第7单元 文件与异常处理

## 7.2 异常处理

### 1. 核心概念

想象一下，你正在玩一个积木搭建游戏。你按照图纸，一步步地搭建一个城堡。突然，你发现有一块关键的积木是坏的，或者图纸上的步骤写错了。这时，整个搭建过程就会卡住，甚至可能导致已经搭好的部分倒塌。

在Python程序运行的过程中，也会发生类似的情况。比如，程序要求用户输入一个数字，但用户却输入了字母；或者程序尝试打开一个不存在的文件。这些“意外”情况，在编程中被称为**异常**。

**异常处理**，就像是为你的程序准备的一套“应急预案”。它允许你预测到程序可能会在哪些地方“摔倒”，并提前在那些地方铺好“安全垫”。当异常发生时，程序不会直接崩溃并显示一堆晦涩难懂的英文错误信息，而是会执行你预先设定好的“安全垫”代码，优雅地处理这个错误，并尝试让程序继续运行下去，或者至少给用户一个友好的提示。

### 2. 关键定义

- **异常（Exception）**：程序运行过程中发生的、会中断正常执行流程的错误事件。例如，`ZeroDivisionError`（除零错误）、`FileNotFoundError`（文件未找到错误）、`ValueError`（值错误）。
- **`try...except` 语句**：Python中用于处理异常的核心结构。`try` 块中放置可能出错的代码，`except` 块中放置当特定异常发生时要执行的“补救”代码。
- **`else` 子句**：可选的子句，放在所有 `except` 块之后。如果 `try` 块中的代码没有引发任何异常，则会执行 `else` 块中的代码。
- **`finally` 子句**：可选的子句，放在最后。无论 `try` 块中是否发生异常，`finally` 块中的代码**都会被执行**。它通常用于执行清理工作，例如关闭文件或释放资源。
- **抛出异常（Raise an Exception）**：使用 `raise` 关键字主动触发一个异常。这在你编写函数时，发现输入参数不合法时非常有用。

### 3. 示例与应用

**生活例子1：ATM取款机**

你去ATM取钱，输入了取款金额1000元。
- **正常流程（`try`）**：ATM机检查你的余额，发现余额充足，于是吐出现金。
- **异常情况（`except`）**：ATM机检查你的余额，发现只有500元。这时，它不会崩溃，而是会执行“余额不足”的异常处理代码，在屏幕上显示“余额不足”，并让你重新输入。
- **最终流程（`finally`）**：无论取款成功还是失败，ATM机最后都会打印一张凭条，并退出你的银行卡。

**应用场景2：读取配置文件**

你的程序启动时需要读取一个名为 `config.txt` 的配置文件。
```python
try:
    # 尝试打开并读取配置文件
    with open('config.txt', 'r') as f:
        config_data = f.read()
        print("配置文件加载成功")
except FileNotFoundError:
    # 如果文件不存在，使用默认配置
    print("配置文件未找到，将使用默认设置")
    config_data = "default_setting"
except PermissionError:
    # 如果没有权限读取文件，给出提示
    print("没有权限读取配置文件，请检查文件权限")
    config_data = "default_setting"
else:
    # 如果文件成功读取，可以执行一些后续操作
    print("正在应用配置...")
finally:
    # 无论结果如何，都打印一条结束信息
    print("配置加载过程结束")
```
这个例子展示了如何针对不同类型的异常（文件不存在、无权限）给出不同的处理方案，并确保程序不会因为一个文件问题而完全崩溃。

### 4. 常见误区

- **误区一：过度使用 `except`，捕获所有异常**
  很多初学者喜欢写 `except:` 或者 `except Exception:` 来捕获所有可能的异常。这虽然能防止程序崩溃，但会掩盖真正的问题。例如，一个 `ZeroDivisionError` 和 `ValueError` 需要完全不同的处理逻辑。不加区分地捕获所有异常，会让调试变得非常困难，因为你不知道程序到底哪里出错了。**最佳实践是只捕获你预料到并能够处理的特定异常类型。**

- **误区二：`try` 块中包含过多代码**
  有些同学会把一大段可能出错的代码都放在一个 `try` 块里。这会导致当异常发生时，你很难定位到底是哪一行代码引发了问题。**一个好的习惯是，将可能出错的、粒度较小的代码块放在 `try` 中，并针对性地捕获异常。**
